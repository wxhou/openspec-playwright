Run Playwright E2E verification for an OpenSpec change. This skill reads the specs generated by `/opsx:propose` and uses Playwright's three-agent pipeline (Planner → Generator → Healer) to validate that the implementation works end-to-end.

**Prerequisites:** Playwright installed, agents initialized, dev server running (or started automatically).

## Step 1: Identify the Change

If no change name is provided, run `openspec list --json` and show available changes. Use **AskUserQuestion tool** to prompt the user to select. **Do NOT guess or auto-select.**

## Step 2: Verify Prerequisites

Check that Playwright agents exist at `.github/` and Playwright MCP is configured. If missing, guide the user to run `openspec-pw init` first.

## Step 3: Read OpenSpec Specs

Read all files from `openspec/changes/<change-name>/specs/*.md`. These are the requirements that E2E tests must cover. Extract every functional requirement, acceptance criterion, and expected behavior.

## Step 4: Run Planner Agent

Generate `openspec/changes/<change-name>/specs/playwright/test-plan.md` by:
- Listing each functional requirement as a test case
- Including expected behaviors, user interactions, and error conditions
- Following the format of any existing test plans in `specs/playwright/`

## Step 5: Run Generator Agent

Create `tests/playwright/<change-name>.spec.ts` by:
- Reading `tests/playwright/seed.spec.ts` as the template format
- Generating Playwright tests covering all items in the test plan
- Using page object patterns and descriptive test names
- Ensuring selectors and assertions are specific and robust

## Step 6: Run Healer Agent (Auto-Heal Loop)

Execute tests with `npx playwright test tests/playwright/<change-name>.spec.ts --reporter=list`:
- If tests pass → record success
- If tests fail → analyze the failure using Playwright MCP tools to inspect the current UI state
- Find equivalent selectors or fix broken assertions
- Patch the test file and re-run
- Continue until all pass or **max 3 healing attempts** are reached

## Step 7: Generate Report

Write `openspec/reports/playwright-e2e-<change-name>-<timestamp>.md` with the verification results.

**Guardrails**

- Always read specs from `openspec/changes/<name>/specs/` as the source of truth
- Do not generate tests that contradict the specs
- Use Playwright MCP tools to inspect live UI state when healing, not guess
- Cap auto-heal attempts at 3 to prevent infinite loops
- If dev server is not running, prompt the user to start it before proceeding
- Write test output to the designated directories, do not overwrite unrelated files
- If no change is specified, always ask the user to select rather than guessing

**Fluid Workflow Integration**

This skill validates the output of `/opsx:apply`. It is typically invoked after `/opsx:verify` (static verification) to add dynamic E2E coverage. Run this skill independently or as a follow-up to static verification.

- **Before:** `/opsx:propose` (generates specs) → `/opsx:apply` (implements)
- **This skill:** `/openspec-e2e` (validates E2E behavior)
- **After:** `/opsx:archive` (if all verifications pass)

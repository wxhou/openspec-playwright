核心特性

为什么选择 OpenSpec Playwright？

专为 OpenSpec 规范驱动开发设计，AI 全流程自动化，测试不再成为交付瓶颈。

🎯
规范驱动测试生成
读取 OpenSpec 规范文档，自动提取路由、表单、验证规则，生成对应 Playwright 测试用例。不再需要手动编写和维护测试。
🤖
Healer 自愈机制
选择器失效时 AI 自动诊断并修复。区分测试 Bug 和 App Bug，三阶段修复流程：分类 → 修复 → 升级。
👁️
真实浏览器探索
通过 gstack /browse 遍历每个路由，snapshot + @ref 精确定位元素。Token 高效，状态持久，可视化操作。
🔒
规范即文档
测试与 OpenSpec 规范严格对齐。验证通过意味着规范得到实现，规范变更自动驱动测试更新。
📊
完整测试报告
自动生成 Markdown 报告，包含测试结果、截图、失败分类（App Bug / Test Bug / RAFT）。可追溯、可分享。
⚡
Claude Code 原生集成
作为 Claude Code skill 提供，一个 /opsx:e2e <name> 命令完成从规范到报告的全流程。
工作流

9 步端到端闭环

严格按顺序执行，每步有明确产出。反馈循环确保问题不过夜。

1
探索与提案 必需
现有项目先探索（/opsx:explore）再写 proposal；新项目直接生成 proposal + scenarios。
2
产品与架构评审 按需触发
/office-hours、/plan-ceo-review、/plan-eng-review。战略和架构影响评审。
3
设计审查 必需
在实现前进行设计评审，确保方案合理。评审通过后开始实现。
4
实现 必需
执行 /opsx:apply 进行实现 → lint + typecheck 通过才算成功。
5
自审 必需
/opsx:verify /design-review /review 自审实现代码，确保质量。
6
E2E 测试生成与执行 必需
/opsx:e2e 生成测试 → /browse 探索 DOM → Healer 修复 → /qa 真实浏览器验证。
7
验证通过后归档 必需
永久归档，更新 specs/。
8
发布 必需
/ship、/land-and-deploy 或 /canary。选择合适的发布策略。
9
迭代回顾 按需触发
评审本次迭代，总结经验教训。/retro
反馈循环
测试失败（App Bug）
→ 回到 步骤 4 修复 → 重新测试
发现架构问题
→ 回到 步骤 2 重新评审 → 步骤 4 修复
Proposal 需调整
→ 回到 步骤 1 重新提案
评审不通过
→ 回到对应步骤重新处理
快速开始

5 分钟上手

已有 Node.js 环境？一行命令安装，立即体验。

安装步骤

STEP 1
安装 CLI
npm install -g openspec-playwright
STEP 2
安装 gstack
git clone --single-branch --depth 1 https://github.com/garrytan/gstack.git ~/.claude/skills/gstack && cd ~/.claude/skills/gstack && ./setup
STEP 3
初始化 OpenSpec
npm install -g @fission-ai/openspec && openspec init
STEP 4
安装 Playwright MCP
claude mcp add playwright npx @playwright/mcp@latest
STEP 5
初始化 E2E
openspec-pw init && npx playwright install --with-deps

运行第一个 E2E 测试

# 在 OpenSpec 项目中，运行： /opsx:e2e <change-name> # Claude Code 自动完成： # 1. 读取规范 → 2. 检测认证 → 3. 探索真实 DOM # 4. 生成测试 → 5. 执行测试 → 6. Healer 修复 # 7. 生成报告

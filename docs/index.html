核心特性

为什么选择
OpenSpec Playwright？

专为 OpenSpec 规范驱动开发设计，AI 全流程自动化，测试不再成为交付瓶颈。

规范驱动
规范驱动测试生成
读取 OpenSpec 规范文档，自动提取路由、表单、验证规则，生成对应 Playwright 测试用例。不再需要手动编写和维护测试。
规范即文档
规范变更，测试即更新
测试与 OpenSpec 规范严格对齐。验证通过意味着规范得到实现，规范变更自动驱动测试更新。
自愈机制
Healer 自愈
选择器失效时 AI 自动诊断并修复。区分测试 Bug 和 App Bug，三阶段修复流程：分类 → 修复 → 升级。
浏览器探索
真实浏览器探索
通过 gstack /browse 遍历每个路由，snapshot + @ref 精确定位元素。Token 高效，状态持久，可视化操作。
测试报告
完整测试报告
自动生成 Markdown 报告，包含测试结果、截图、失败分类（App Bug / Test Bug / RAFT）。可追溯、可分享。
Claude Code
Claude Code 原生集成
作为 Claude Code skill 提供，一个 /opsx:e2e <name> 命令完成从规范到报告的全流程。

gstack

/browse 探索 + /qa 验证
OpenSpec CLI
npm install -g @fission-ai/openspec && openspec init
/opsx:propose、/opsx:apply 等命令
openspec-playwright
npm install -g openspec-playwright && openspec-pw init
/opsx:e2e 命令 + 测试模板
Playwright MCP
claude mcp add playwright npx @playwright/mcp@latest
测试执行 + Healer 自愈
1
探索与提案 必需
现有项目先探索（/opsx:explore）再写 proposal；新项目直接生成 proposal + scenarios。方向不明确时可用 /office-hours 做创意验证。
opsx:explore opsx:propose office-hours
2
产品与架构评审 按需
/plan-ceo-review、/plan-eng-review。战略和架构影响评审。
plan-ceo-review plan-eng-review
3
设计审查 必需
在实现前进行设计评审，确保方案合理。评审通过后开始实现。
plan-design-review
4
实现 必需
执行 /opsx:apply 进行实现 → lint + typecheck 通过才算成功。
opsx:apply
5
自审 必需
/opsx:verify /design-review /review 自审实现代码，确保质量。设计审查后执行两步式 CSS 审计：1) grep 提取间距值对比基准；2) 检查 margin hack（margin 和 > 基准 2 倍 → 改为 gap/grid 控制）。
opsx:verify design-review review CSS审计
6
E2E 测试生成与执行 必需
/opsx:e2e 生成测试 → /browse 探索 DOM → Healer 修复（批量修复后 --only-changed 一次性验证）→ /qa 真实浏览器验证。
opsx:e2e browse qa Playwright MCP
7
验证通过后归档 必需
永久归档，更新 specs/。
opsx:archive
8
发布 按需
/ship、/land-and-deploy 或 /canary。选择合适的发布策略。
ship land-and-deploy canary
9
迭代回顾 按需
评审本次迭代，总结经验教训。/retro
retro

✕
测试失败（App Bug）
→ 回到 步骤 4 修复代码 → 重新执行 E2E
△

→ 回到 步骤 2 重新评审 → 步骤 4 修复
○
Proposal 需调整
→ 回到 步骤 1 重新提案
◻
评审不通过
→ 回到对应步骤重新处理
